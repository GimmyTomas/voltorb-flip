How the Solver Works

1. Introduction

Voltorb Flip is a minigame from Pokémon HeartGold/SoulSilver that combines elements of Minesweeper and Picross. The goal is to flip all panels containing multipliers (2s and 3s) while avoiding Voltorbs (bombs).

This solver uses Bayesian inference and constraint enumeration to calculate the exact probability that each hidden panel contains a Voltorb, then uses iterative deepening minimax (expectimax) to find the move that maximizes the probability of winning the game.

2. Why This Solver Is Different

Most Voltorb Flip solvers share two fundamental limitations that lead to suboptimal play:

Limitation 1: Ignoring Board Generation Mechanics

Other solvers treat all constraint-satisfying boards as equally likely. In reality, the game selects from 10 predefined board types per level, each with different panel compositions and legality constraints. Because some types produce more valid boards than others, they are not equally probable given the observed hints.

This solver uses Bayesian inference to compute P(type | board) for each of the 10 types, weighting each type by its N_accepted count. The result is more accurate per-panel probabilities that reflect the true generation process.

Limitation 2: Greedy Strategy

Other solvers recommend the panel with the lowest Voltorb probability — a greedy, one-step-ahead heuristic. This ignores how the current choice affects future moves: flipping a slightly riskier panel now may reveal information that makes the rest of the game much safer.

This solver uses iterative deepening minimax (expectimax) to search the full game tree and find the move that maximizes the overall probability of winning, not just surviving the next flip.

3. Board Types & Generation

Board Type Parameters

Each level has 10 possible board types (T0-T9). Each type defines:

// Board type parameters n0 = number of Voltorbs n2 = number of 2s n3 = number of 3s n1 = 25 - n0 - n2 - n3 // number of 1s (derived) // Legality constraints maxTotalFree = max multipliers in "free" rows/columns combined maxRowFree = max multipliers in any single free row/column

A row or column is "free" if its hint shows 0 Voltorbs. These constraints prevent boards that would be too easy to solve.

Example: Level 1, Type 0

n0=6, n2=3, n3=1, maxTotalFree=3, maxRowFree=3

This board has 6 Voltorbs, 3 Twos, 1 Three, and 15 Ones (25-6-3-1=15).
Total sum = 15 + 2×3 + 3×1 = 24

N_accepted: Valid Board Count

For each board type, N_accepted is the pre-computed count of all valid board configurations that satisfy the legality constraints. This number is used as a normalization factor in probability calculations.

// Example N_accepted values for Level 1 T0: 1,732,660,000 valid boards T1: 81,056,200 valid boards T2: 1,407,934,200 valid boards ...

Random Board Generation

The game generates boards using rejection sampling:

  1. Select a random board type uniformly from T0-T9
  2. Generate a random arrangement of n0, n1, n2, n3 panels
  3. Check legality constraints (free row/column limits)
  4. If illegal, reject and try again; if legal, use this board

4. Compatible Board Enumeration

Given the current hints and any revealed panels, the solver finds all possible complete boards. This is a two-phase constraint satisfaction process:

Phase 1: Voltorb Position Enumeration

Generate all valid placements of Voltorbs that satisfy the row and column hints.

for each row i: // Place exactly hint[i].voltorbCount Voltorbs in row i for each combination of positions: if column constraints still satisfiable: recurse to next row when all rows filled: if column voltorb counts match exactly: if no conflict with revealed panels: yield this voltorb configuration

Phase 2: Fill Non-Voltorb Positions

For each valid Voltorb configuration and each compatible board type, fill the remaining positions with 1s, 2s, and 3s according to the type's requirements.

for each voltorb configuration: for each compatible board type: // Must place exactly n2 twos and n3 threes for each unknown position: try placing 1, 2, or 3 if constraints still satisfiable: recurse to next position when complete: if legal (free row/col constraints): add to compatible boards

Constraint Pruning

At each step, the solver prunes branches early by checking:

  • Row/column sum doesn't exceed the hint
  • Voltorb count doesn't exceed the hint
  • Remaining positions can achieve required sum (min=1, max=3 per non-Voltorb)
  • Enough positions remain to place required 2s and 3s

5. Probability Calculation (Bayesian Inference)

The Key Insight

Different board types have different N_accepted values, meaning some types are more likely than others given the same observed hints. We use Bayes' theorem to account for this.

Important: Board-to-Type Assignment

Multiple board types can have the same (n0, n2, n3) but different legality constraints. For correct Bayesian inference, each compatible board must be counted in all types it is legal for, not just the first match.

Example: Level 2 has Type 0 and Type 5 both with (n0=7, n2=1, n3=3), but Type 0 has maxTotalFree=3 while Type 5 has maxTotalFree=2. A board satisfying the stricter Type 5 constraint is legal for both types and contributes to both types' probability mass.

Step 1: Calculate P(type | board)

For each board type, compute the probability it generated the current board:

P(board | type) = compatible_count[type] / N_accepted[type]
Probability of seeing this board if it came from this type

Then normalize across all types (using uniform prior P(type) = 1/10):

P(type | board) = P(board | type) / Σt P(board | typet)

Step 2: Calculate P(panel = value | type, board)

For each board type, count how many compatible boards have each value at this panel:

P(value | type, board) = count_with_value[type] / compatible_count[type]
Fraction of compatible boards of this type where the panel has this value

Step 3: Calculate P(panel = value | board)

Combine using the law of total probability:

P(value | board) = Σtype P(value | type, board) × P(type | board)

This marginalizes over the unknown board type, weighting each type's contribution by its posterior probability.

Worked Example

Given: Level 2, sum hints totaling 25, 7 Voltorbs

Compatible types: T0 and T5 (both have n0=7, n2=1, n3=3)

These types share the same panel composition but have different legality constraints: T0 has maxTotalFree=3, while T5 has the stricter maxTotalFree=2.

Suppose after enumeration, 150 boards are compatible with each type:

T0: 150 compatible boards, N_accepted = 3,285,100,800 T5: 150 compatible boards, N_accepted = 2,684,683,200

Calculate P(board | type) for each:

T0: 150 / 3,285,100,800 = 4.57 × 10-8 T5: 150 / 2,684,683,200 = 5.59 × 10-8

After normalization:

P(T0 | board) = 4.57 / (4.57 + 5.59) = 45.0% P(T5 | board) = 5.59 / (4.57 + 5.59) = 55.0%

Even though both types have the same composition and the same compatible board count, T5 gets higher posterior weight because its stricter constraint (maxTotalFree=2) means fewer total valid boards — making each compatible board more “surprising” and thus more likely.

For a panel where:

  • In T0 boards: 80% Voltorb, 10% One, 8% Two, 2% Three
  • In T5 boards: 70% Voltorb, 15% One, 10% Two, 5% Three

Final P(Voltorb) = 0.80×0.450 + 0.70×0.550 = 74.5%

6. Solver Algorithm

Goal

Find the panel that maximizes the probability of eventually winning the game (revealing all multipliers without hitting a Voltorb).

Win Probability Definition

The optimal win probability for a board state S is defined recursively:

f(S) = 1   if all multipliers (2s and 3s) are revealed
f(S) = 0   if a Voltorb has been revealed
f(S) = maxpanel { Σv P(panel=v | S) × f(S′) }   otherwise
where S′ is the state after revealing the panel as value v, and v ranges over {1, 2, 3} (Voltorb outcomes contribute f=0)

The optimal move is the panel that achieves this maximum. This is an expectimax formulation: the solver maximizes over panel choices and takes expectations over the random revealed value.

Free Panel Detection

A panel is "free" (guaranteed safe) if it is never a Voltorb in any compatible board. The solver always recommends free panels first.

function findFreePanels(compatibleBoards): for each unknown panel: isSafe = true for each compatible board: if panel is Voltorb in this board: isSafe = false break if isSafe: freePanels.add(panel) return freePanels

Iterative Deepening

Computing the exact win probability requires exploring the full game tree, which can be intractable for complex boards. The solver uses iterative deepening to provide anytime results that improve with more computation:

function iterativeDeepening(board, timeout): for depth = 1 to maxDepth: memo = new Map() // fresh memo each depth result = depthLimitedSearch(board, depth, memo) yield result if result.isExact or elapsed > timeout: break

Each depth iteration creates a fresh memoization table. This prevents stale results from shallower searches from limiting deeper explorations. The solver yields progress at each depth, so the UI can display the best result found so far.

Depth-Limited Search

At each depth, the solver performs a bounded expectimax search:

function depthLimitedSearch(state, depthLimit, memo): if isWon(state): return (winProb=1.0, exact=true) if depthLimit == 0: return (winProb=heuristic(state), exact=false) if memo.has(state): return memo.get(state) // Free panels: recurse without spending depth if freePanel exists: reveal freePanel, recurse at same depthLimit bestWinProb = 0 for panel in orderedUnknownPanels(state): // Upper-bound pruning if (1 - P(Voltorb)) ≤ bestWinProb: continue expected = Σv∈{1,2,3} P(panel=v) × depthLimitedSearch(reveal(state,panel,v), depthLimit-1) bestWinProb = max(bestWinProb, expected) memo.set(state, result) return result

Heuristic Evaluation

When the depth limit is reached, the solver estimates win probability instead of returning an arbitrary value:

  1. Count the multipliers (2s and 3s) still needing to be revealed
  2. Collect the Voltorb probability of each risky panel
  3. Sort risky panels by safety (lowest Voltorb probability first)
  4. Return the product of survival probabilities for the N safest panels, where N is the number of multipliers still needed
function heuristicEval(state): multipliersNeeded = countRemainingMultipliers(state) if multipliersNeeded == 0: return 1.0 voltorbProbs = [P(Voltorb) for each risky panel] voltorbProbs.sort() // lowest risk first survival = 1.0 for i = 0 to min(voltorbProbs.length, multipliersNeeded): survival ×= (1 - voltorbProbs[i]) return survival

Memoization

The search space is a DAG (different move orders can reach the same state), so memoization is essential. The JavaScript solver uses compactKey() to produce a collision-free string key:

function compactKey(): key = level + 25 panel digits // each panel value shifted to single digit return key // unique string, used as Map key

A fresh Map is created for each depth iteration. This avoids stale entries from shallower searches corrupting deeper ones.

Optimizations

  • Move ordering: Panels are sorted by Voltorb probability (safest first), so good moves are evaluated early and pruning is more effective
  • Upper-bound pruning: A panel's best possible win probability is 1 - P(Voltorb). If this is ≤ the current best, the panel is skipped entirely
  • Free panel short-circuit: Guaranteed-safe panels are revealed immediately without spending search depth, since they are always optimal
  • Useless panel pruning: Panels that have no multiplier potential (can only be 1 or Voltorb across all compatible boards) are skipped entirely in the search, free panel detection, and heuristic evaluation, since flipping them can never make progress toward winning

Understanding Results

The solver reports a win probability and a search depth for each recommendation:

  • "Exact" — the full game tree was explored; the win probability is provably correct and the suggested move is provably optimal
  • "Depth N (approximate)" — the solver searched N moves ahead and used heuristic evaluation at the frontier; the result is a lower bound on the true win probability and improves with deeper search

The default timeout is 30 seconds. For most boards, the solver reaches an exact solution well within this limit.

Approximation Bounds

When the search is depth-limited (not exact), the solver displays a range (e.g., “23.2% ~ 38.5%”) representing lower and upper bounds on the true optimal win probability V*(s).

Lower Bound (Conjecture)

Claim: At fixed depth d, the depth-limited win probability Vd(s) ≤ V*(s).

Proof structure: If the heuristic h(s) ≤ V*(s) at all leaf states, then Vd(s) ≤ V*(s) at the root. This follows by standard induction on the expectimax tree: both expectation (at chance nodes) and max (at decision nodes) preserve the ≤ relation.

Why h(s) ≤ V*(s) (strong conjecture, not fully rigorous):

The heuristic computes h(s) = ∏(1 − pvoltorb[i]) for the safest N panels with multiplier potential, where N = multipliersNeeded. Three factors make this ≤ V*(s):

  1. Overcounting: multipliersNeeded counts positions where any compatible board has a multiplier. The actual board has strictly fewer multipliers than this count whenever boards disagree on multiplier placement.
  2. No information gain: The heuristic uses fixed marginal probabilities. The optimal strategy adapts after each reveal — learning the board narrows possibilities and can reveal which panels are safe.
  3. No early winning: The heuristic requires surviving all N panels. In practice, the game often ends earlier (the actual board may have fewer multipliers, or remaining panels become provably safe).
Caveat: The product of marginals actually overestimates the joint survival probability (due to negative correlation in voltorb placement). However, extensive empirical testing confirms that the three factors above dominate this overestimate.

Upper Bound Estimator

Formula: hupper(s) = ∏(1 − pvoltorb[i]) for the safest Mmin panels with multiplier potential, where Mmin = minimum unrevealed multipliers across all compatible boards. If Mmin = 0 (some board has all multipliers already revealed), hupper = 1.0 (empty product).

This is an estimator that usually overestimates V*(s), making it a practical upper bound in most cases. However, counterexamples exist where the true win probability slightly exceeds hupper(s), so it is not a rigorous upper bound.

The estimator uses the most optimistic panel count (Mmin — the fewest multipliers the player could need to survive) combined with the safest voltorb probabilities, which tends to produce a value above V*(s). It is more informative and converges faster than the trivial h=1.0 fallback used at depth-limited leaves.

When the search is exact (fully explored), the estimator is not used and the computed value equals V*(s) exactly.

7. Implementation Notes

Three Implementations

Three solver implementations exist:

  • C++ solver — iterative deepening minimax with Zobrist hashing and fixed-size transposition table (CLI applications)
  • JavaScript web solver — iterative deepening minimax with compactKey() and Map for memoization
  • WASM solver — the C++ solver compiled to WebAssembly via Emscripten, running at near-native speed in the browser. Use the engine toggle in solver settings to switch between JS and WASM.

All three solvers use the same algorithm and produce equivalent results. The C++ and WASM solvers share the same codebase and use Zobrist hashing with a transposition table, while JavaScript builds a collision-free string key (compactKey()) and stores results in a standard Map. Both the JS and WASM solvers report per-depth progress, allowing the UI to show improving results in real time.

Bayesian Inference (All Implementations)

All solvers use two-stage Bayesian inference:
  1. Calculate P(type | board) using compatible counts and N_accepted
  2. Calculate P(value | type, board) from compatible boards per type
  3. Combine: P(value | board) = Σ P(value | type, board) × P(type | board)

Critical detail: Each compatible board is checked for legality against all types with matching (n0, n2, n3) and counted in every type it satisfies. This ensures correct probability weighting when multiple types share the same panel counts but have different legality constraints.

Performance Considerations

  • Compatible board enumeration is capped at 500,000 boards (matching the C++ solver)
  • Constraint pruning significantly reduces the search space
  • BigInt is used for N_accepted values (can exceed 1012)
  • Iterative deepening with a 5-second timeout provides anytime results
  • Memoization via compactKey() (JS) or Zobrist hashing (C++) avoids recomputing the same board states
